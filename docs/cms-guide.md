# CMS на Wagtail

## Роль Wagtail в проекте

Wagtail в этом проекте — это **самостоятельная редакторская ветка сайта**, а не основное хранилище изданий.

Основной прототип отвечает за:
- издания;
- поиск;
- коллекции;
- рекомендации.

Wagtail отвечает за:
- объявления и новости;
- редакторские страницы;
- дополнительные самостоятельные разделы сайта.

## Базовая структура CMS

Команда `bootstrap_wagtail` создаёт и поддерживает базовое дерево:

- `CMS-раздел` (`HomePage`)
  - `Объявления` (`AnnouncementIndexPage`)
    - страницы объявлений (`AnnouncementPage`)
  - `Страницы редакции` (`ContentIndexPage`)
    - свободные редакторские страницы (`ContentPage`)

Дополнительно создаются стартовые заготовки:
- `О редакционном разделе`
- `Как публиковать материалы`
- `Контакты редакции`
- приветственное объявление

## Когда запускать `bootstrap_wagtail`

```bash
uv run python manage.py bootstrap_wagtail
```

Запускайте:
- при первом старте проекта;
- после смены БД;
- после существенного изменения CMS-структуры;
- если нужно безопасно досинхронизировать дерево страниц.

Команда не должна без необходимости дублировать уже существующие страницы: она старается переиспользовать имеющиеся узлы.

## Типы CMS-страниц

### `HomePage`
Корень CMS-ветки.

Используется как точка входа в редакционный раздел.

### `AnnouncementIndexPage`
Каталог объявлений.

Используется для:
- новостей проекта;
- анонсов;
- служебных публикаций для пользователей и редакторов.

### `AnnouncementPage`
Отдельная страница объявления.

Поддерживает:
- дату публикации;
- закрепление;
- summary;
- гибкое тело страницы через `StreamField`.

### `ContentIndexPage`
Каталог свободных редакторских страниц.

### `ContentPage`
Универсальная CMS-страница для:
- инструкций;
- разделов справки;
- регламентов;
- контактной информации;
- будущих самостоятельных разделов сайта.

## StreamField и свободная структура контента

`AnnouncementPage` и `ContentPage` используют `StreamField`.

Доступные типы блоков включают:
- подзаголовки;
- текстовые блоки;
- изображения;
- документы;
- embed;
- callout-блоки;
- цитаты;
- кнопки;
- карточки.

Это даёт редакторам достаточно свободы без необходимости сразу заводить новый тип страницы под каждый новый сценарий.

## Когда нужен новый page type, а когда достаточно `ContentPage`

### Достаточно `ContentPage`, если
- страница контентная;
- нужна свободная композиция из блоков;
- нет особой бизнес-логики.

### Нужен отдельный тип страницы, если
- у страницы есть своя доменная логика;
- нужны особые поля и отдельные списки;
- нужен специфический шаблон или workflow;
- появляется отдельный новый раздел сайта с собственным behaviour.

## Безопасная кастомизация Wagtail admin

Рекомендуемый подход:
- использовать брендинг и лёгкие CSS-акценты;
- не переопределять базовые шаблоны admin без крайней необходимости;
- не ломать layout через агрессивные глобальные селекторы.

Если Wagtail admin начинает “разъезжаться”, сначала нужно вернуть дефолтный layout, а потом стилизовать интерфейс очень точечно.

## Как расширять CMS в будущем

### Вариант 1. Новые свободные страницы
Просто создавать новые `ContentPage` в существующем каталоге `Страницы редакции`.

### Вариант 2. Новый самостоятельный раздел
Например:
- FAQ;
- методические материалы;
- редакционный блог.

Лучший путь:
1. создать новый тип index-page;
2. создать дочерний page type;
3. добавить их в `bootstrap_wagtail`;
4. добавить ссылки в публичную навигацию сайта.

### Вариант 3. Медиа- и документные подборки
Можно развивать StreamField-блоки или вводить отдельные page-models под специальные сценарии.
