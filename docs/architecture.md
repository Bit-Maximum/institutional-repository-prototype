# Архитектура проекта

## Общая модель

Проект построен как **модульный монолит** на Django. Доменные сценарии вынесены в отдельные приложения, но работают в рамках одного deployable-приложения и одной общей схемы данных.

Такой подход выбран как компромисс между:
- скоростью разработки прототипа;
- контролем над связями между сущностями;
- возможностью расширять проект без дробления на избыточное число сервисов.

## Модули

### `apps.core`
Базовые страницы и общая инфраструктура:
- главная страница;
- health-check endpoints;
- общие admin mixins и dashboard;
- статистические представления для админки.

### `apps.users`
Пользовательский контур:
- регистрация и аутентификация;
- интеграция с `django-allauth`;
- предпочтения пользователя;
- сигналы и адаптеры.

### `apps.publications`
Предметное ядро репозитория:
- издания;
- справочники (авторы, издатели, языки, ключевые слова и т.д.);
- черновики и редактор публикаций;
- генерация и хранение превью;
- пользовательские взаимодействия с изданиями.

### `apps.collections_app`
Коллекции пользователей:
- карточки коллекций;
- страница коллекции;
- управление своими коллекциями;
- реакции и share-сценарии.

### `apps.ingestion`
Предобработка исходных файлов:
- извлечение текста;
- определение поддерживаемых форматов;
- prefill метаданных из файла.

### `apps.search`
Поисковый слой:
- формы поиска;
- keyword / semantic / hybrid сервисы;
- поисковая история;
- рекомендации;
- benchmark.

### `apps.vector_store`
Изоляция работы с Milvus:
- создание и валидация коллекции;
- bulk reindex;
- warmup моделей и vector-store;
- конфигурация retrieval pipeline.

### `apps.cms`
CMS-ветка на Wagtail:
- объявления;
- редакторские страницы;
- bootstrap базового дерева CMS;
- навигационные данные для публичного сайта.

### `apps.ui`
Публичный UI-слой:
- глобальные стили интерфейса;
- тёмная / светлая / системная тема;
- язык интерфейса;
- инъекция UI-состояния в шаблоны.

## Данные и границы хранения

### PostgreSQL
Хранит:
- основные доменные сущности;
- связи многие-ко-многим;
- черновики и опубликованные записи;
- историю поиска;
- рекомендации и взаимодействия пользователей;
- настройки публичного интерфейса.

### Milvus
Хранит:
- dense/sparse представления фрагментов документов;
- данные, необходимые для semantic и hybrid retrieval.

### Media storage
Хранит:
- файлы изданий;
- ручные и сгенерированные превью;
- CMS-документы и медиа.

## Логика публикации изданий

Сценарий редактора устроен так:
1. создаётся черновик или новая запись;
2. файл может быть загружен сразу, позже или не загружаться вообще;
3. часть метаданных предзаполняется из файла;
4. прогресс-бар показывает, сколько ключевых полей уже закрыто;
5. запись можно сохранить как черновик;
6. публикация требует обязательного минимума полей.

Ключевые поля для публикации:
- название;
- файл или внешняя ссылка;
- год издания;
- язык;
- подтип издания;
- авторы;
- аннотация / описание.

## Превью изданий

Система поддерживает два режима:
- **ручное превью**, загруженное редактором;
- **автогенерация** из файла.

Приоритет у ручного превью выше. Автогенерация используется как fallback.

## Поисковая архитектура

### Keyword search
Основан на реляционных данных и текстовых полях.

### Semantic search
Работает поверх chunk-индекса в Milvus.

### Hybrid search
Смешивает традиционный и семантический контур с настраиваемыми коэффициентами и порогами.

### Explain layer
У результатов поиска выводится вторичный блок с пояснением:
- raw score;
- final score;
- источник совпадения;
- фрагмент текста;
- этап ранжирования.

## CMS-архитектура

Wagtail-ветка отделена от доменной логики изданий и коллекций.

Основная идея:
- основной сайт живёт в пользовательском UI прототипа;
- Wagtail — это независимый редакторский слой для объявлений и дополнительных страниц;
- `bootstrap_wagtail` подготавливает дерево страниц так, чтобы редакторам не приходилось собирать CMS с нуля вручную.

## Почему это модульный монолит, а не набор микросервисов

Для прототипа это даёт преимущества:
- проще поддерживать целостность модели данных;
- меньше интеграционных накладных расходов;
- легче обеспечивать единый UI и единые сценарии пользователей;
- быстрее вносить изменения в предметную модель и интерфейс.

В дальнейшем при необходимости можно выделять в отдельные сервисы:
- индексацию;
- recommendations pipeline;
- media / preview processing.
